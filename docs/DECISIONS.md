# DECISIONS.md — PetLand v20

## تصمیمات معماری و فنی

این فایل تصمیماتی را ثبت می‌کند که از سورس کد قابل استنتاج هستند. دلایل «چرا» در صورت عدم وجود مستندات با `[استنتاج]` علامت‌گذاری شده‌اند.

---

### ADR-001: Bale به‌جای Telegram
- **تصمیم:** استفاده از پیام‌رسان Bale (tapi.bale.ai)
- **دلیل:** بازار هدف ایران است. Bale پیام‌رسان ایرانی است و محدودیت‌های فیلترینگ را ندارد.
- **پیامد:** API مشابه Telegram ولی جداگانه — `tapi.bale.ai` به‌جای `api.telegram.org`

---

### ADR-002: Long Polling به‌جای Webhook
- **تصمیم:** استفاده از long polling برای دریافت آپدیت‌ها
- **دلیل:** [استنتاج] راحت‌تر برای dev محلی، نیاز به SSL ندارد
- **پیامد:** نیاز به پروسه دائمی — serverless ممکن نیست. فایل `webhook.js` stub باقی مانده

---

### ADR-003: Prisma db:push به‌جای Migrations
- **تصمیم:** استفاده از `prisma db push` بدون migration files
- **دلیل:** [استنتاج] سادگی در مرحله توسعه اولیه
- **پیامد:** تاریخچه تغییرات DB وجود ندارد، rollback ممکن نیست

---

### ADR-004: وضعیت مکالمه در Database
- **تصمیم:** `orderStep`، `adminStep` و فیلدهای temp روی مدل `User` در DB ذخیره می‌شوند
- **دلیل:** [استنتاج] ساده‌ترین راه حل برای persistence بین پیام‌ها — in-memory از restart زنده نمی‌ماند
- **پیامد:** هر پیام ورودی حداقل یک DB read+write انجام می‌دهد
- **توسعه:** `adminStep = "VIEW_MY_ORDERS"` برای bypass کردن admin handler هنگام مشاهده سفارشات شخصی توسط ادمین استفاده می‌شود

---

### ADR-005: Static Product Catalog + Seed
- **تصمیم:** محصولات در `src/data/products.js` به صورت static تعریف شده و با `seed.js` به DB وارد می‌شوند
- **دلیل:** [استنتاج] محصولات به‌ندرت تغییر می‌کنند — نیاز به CMS نیست
- **پیامد:** برای تغییر محصولات باید فایل JS ویرایش و seed مجدد اجرا شود

---

### ADR-006: JavaScript CommonJS به‌جای TypeScript
- **تصمیم:** پروژه با JavaScript ساده (CommonJS) نوشته شده
- **دلیل:** [استنتاج] سرعت توسعه بیشتر، تیم کوچک
- **پیامد:** بدون type safety، بدون interface — نیاز به دقت بیشتر در نام‌گذاری

---

### ADR-007: Liara به‌عنوان بستر استقرار
- **تصمیم:** استقرار روی Liara (PaaS ایرانی)
- **دلیل:** اسکریپت `build` از `PRISMA_ENGINES_MIRROR=https://prisma.storage.iran.liara.space` استفاده می‌کند
- **پیامد:** نیاز به mirror ایران برای دانلود Prisma engines

---

### ADR-008: قیمت‌گذاری cost-plus
- **تصمیم:** قیمت فروش = قیمت خرید + درصد سود (پیش‌فرض 15%)
- **دلیل:** [استنتاج] مدل کسب‌وکار ساده
- **پیامد:** تغییر قیمت فروش نیازمند تغییر `costPrice` یا `profitPercent` در DB است

---

### ADR-009: احراز هویت مبتنی بر Bale ID
- **تصمیم:** هیچ سیستم login/password ندارد — شناسایی بر اساس `baleId` کاربر است
- **دلیل:** کاربران از طریق Bale verified هستند
- **پیامد:** هر کسی که به ربات پیام بدهد، customer می‌شود. امنیت وابسته به امنیت اکانت Bale کاربر است

---

### ADR-011: Inline Keyboard و callback_query برای UI تعاملی
- **تصمیم:** لیست محصولات، سفارشات، فاکتورها و تیکت‌ها به‌جای متن ساده با inline keyboard نمایش داده می‌شوند
- **دلیل:** UX بهتر — کاربر نیازی به تایپ کد ندارد؛ ناوبری با کلیک
- **پیامد:** `index.js` هم `message` و هم `callback_query` را پردازش می‌کند. `handleCallbackQuery` در `router.js` اضافه شد. `answerCallbackQuery` در `bale.js` پیاده‌سازی شد. `inlineKb()` در `menus.js` اضافه شد.
- **callback_data conventions:**
  - `product:CODE` — نمایش محصول
  - `PL-...` — نمایش سفارش کاربر
  - `ordr:UUID` — نمایش فاکتور برای ادمین
  - `rej_more:N` / `shipd_more:N` — pagination فاکتورها
  - `ship:snapp:UUID` / `ship:post:UUID` — انتخاب نوع ارسال
  - `tkt:view:UUID` / `tkt:more:N` — تیکت‌های ادمین
  - `cat:back` / `main:back` — ناوبری برگشت

---

### ADR-012: flow ارسال سفارش (اسنپ / پست)
- **تصمیم:** ادمین هنگام ثبت ارسال، نوع حمل را انتخاب می‌کند (اسنپ یا پست)
- **دلیل:** اطلاعات ارسال برای هر نوع متفاوت است و باید به کاربر اطلاع داده شود
- **پیامد:** دو adminStep جدید: `SHIP_SNAPP` (شماره+پلاک+مدل) و `SHIP_POST` (کد پیگیری). `shipmentInfo` در DB با prefix `اسنپ |` یا `پست |` ذخیره می‌شود.

---

### ADR-010: پرداخت دستی (کارت به کارت / شبا)
- **تصمیم:** پرداخت آنلاین وجود ندارد — کاربر رسید آپلود می‌کند، ادمین تأیید می‌کند
- **دلیل:** [استنتاج] سادگی پیاده‌سازی، عدم نیاز به درگاه پرداخت
- **پیامد:** فرآیند دستی — ادمین باید هر سفارش را بررسی کند
- **توسعه:** `BANK_IBAN` به env و `buildPaymentInfo()` اضافه شد — کاربر می‌تواند با شبا هم واریز کند

---

### ADR-013: سیستم بازاریابی و معرف
- **تصمیم:** هر کاربر یک لینک معرفی اختصاصی دارد (`/start ref_<baleId>`)
- **دلیل:** رشد ارگانیک از طریق معرفی — انگیزه مالی (پورسانت)
- **پیامد:**
  - `referrerId` (self-relation) روی مدل `User` — یک بار هنگام اولین ورود ثبت می‌شود و تغییر نمی‌کند
  - کاربرانی که بدون لینک معرفی وارد شوند، برای همیشه بدون معرف باقی می‌مانند
  - `index.js` پارامتر `/start ref_xxx` را parse کرده و به `getOrCreateUser` پاس می‌دهد
  - `router.js` برای مطابقت `/start ref_xxx` از `startsWith("/start ")` استفاده می‌کند

---

### ADR-014: سیستم کیف پول و پورسانت
- **تصمیم:** 5٪ مبلغ هر فاکتور تایید‌شده به کیف پول معرف واریز می‌شود
- **دلیل:** انگیزه مالی برای بازاریابی — مدل affiliate marketing
- **پیامد:**
  - مدل‌های `Wallet` و `Withdrawal` به schema اضافه شدند
  - `Wallet` یک‌به‌یک با `User` — با `upsert` در اولین نیاز ساخته می‌شود
  - کمیسیون در `approveOrder` (admin.js) هنگام تغییر وضعیت به PACKAGING محاسبه و واریز می‌شود
  - برداشت: حداقل 50,000 تومان، حداکثر 10,000,000 تومان — ادمین باید تایید کند
  - پس از تایید ادمین: موجودی کسر شده و کد رهگیری به کاربر ارسال می‌شود

---

### ADR-016: آدرس ذخیره‌شده در checkout
- **تصمیم:** بعد از اولین سفارش موفق، آدرس ارسال کاربر در مدل `SavedAddress` ذخیره می‌شود (حداکثر ۳ آدرس)
- **دلیل:** کاهش اصطکاک برای سفارش‌های تکراری — کاربر نیازی به وارد کردن مجدد آدرس ندارد
- **پیامد:**
  - مدل `SavedAddress` با فیلدهای `fullName`, `phone`, `province`, `city`, `address`, `postalCode` اضافه شد
  - `tempAddressId String?` روی User برای state نگهداری آدرس انتخاب‌شده در inline menu
  - `startCheckout` ابتدا آدرس‌های ذخیره‌شده را چک می‌کند — اگر وجود داشت inline نمایش می‌دهد
  - تأیید آدرس ذخیره‌شده → مستقیم به `finalizeOrder` (بدون مراحل وارد کردن آدرس)
  - حذف آدرس با دکمه «حذف مشخصات ثبت شده» امکان‌پذیر است
  - `callback_data` conventions: `addr:view:<id>`, `addr:new`
  - `orderStep: "ADDR_CONFIRM"` برای state نگهداری انتخاب آدرس

---

### ADR-017: گیت‌بندی بازاریابی و کیف پول با کد مخفی
- **تصمیم:** دکمه‌های «بازاریابی» و «کیف پول» در منوی اصلی پیش‌فرض مخفی هستند
- **دلیل:** این قابلیت‌ها برای همه کاربران مناسب نیست — فقط بازاریابان باید به آن‌ها دسترسی داشته باشند
- **پیامد:**
  - `marketingEnabled Boolean @default(false)` روی User اضافه شد
  - کاربر با ارسال `MARKETING_ACCESS_CODE` (پیش‌فرض: `petland-vip`) دسترسی پیدا می‌کند
  - `mainMenu()` دکمه‌ها را فقط اگر `user.marketingEnabled === true` باشد نمایش می‌دهد
  - دسترسی یک‌طرفه است — لغو از طریق DB مستقیم

---

### ADR-018: آمار فروش ماهانه در پنل ادمین
- **تصمیم:** آمار فروش به صورت آرشیو ماهانه در مدل `MonthlySalesReport` ذخیره می‌شود
- **دلیل:** نمایش زنده از روی سفارشات برای ماه‌های گذشته کافی است ولی برای آرشیو long-term بهتر است snapshot نگه داشته شود
- **پیامد:**
  - مدل `MonthlySalesReport` با فیلدهای `yearMonth`, `totalRevenue`, `totalProfit`, `totalCommission`, `orderCount`
  - آرشیو هنگام باز کردن صفحه آمار توسط ادمین به‌روز می‌شود (lazy archiving)
  - حداکثر ۶ ماه نگه داشته می‌شود — ماه‌های قدیمی‌تر حذف می‌شوند
  - ماه جاری همیشه live محاسبه می‌شود
  - سود خالص = سود ناخالص منهای مجموع پورسانت‌ها
  - `callback_data` conventions: `stats:YYYY-MM:live`, `stats:YYYY-MM:arch`
- **تصمیم:** صفحاتی که هم لیست inline دارند هم دکمه بازگشت، دو پیام جداگانه می‌فرستند
- **دلیل:** Bale API اجازه ترکیب `inline_keyboard` و `keyboard` در یک پیام را نمی‌دهد
- **پیامد:**
  - پیام اول: متن header + `keyboard` عادی (با دکمه بازگشت) — به عنوان `lastMessageId` track می‌شود
  - پیام دوم: لیست inline — `message_id` آن روی `lastMessageId` override می‌شود تا در ناوبری بعدی حذف شود
  - این الگو در `products.js` (لیست محصولات دسته) و `order.js` (لیست سفارشات) استفاده شده
